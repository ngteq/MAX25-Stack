# MAX25-Stack v2.0.0 — scope and mandatory goals

**Status:** planning · **Predecessor:** [V1.0.0-SCOPE.md](V1.0.0-SCOPE.md)

v2 is the release line where MAX25 becomes **easy to run on a home Linux box** (Raspberry Pi, small homeserver, repurposed PC) — including operators who know Windows but not Unix administration.

## DEV-Levels roadmap (stack-wide)

> **Orientation only:** DEV-Levels are **approximate guidelines** (*ca.*) — items are often pulled forward when dependencies allow. They give general direction, not rigid release gates.

| DEV-Level | Focus (approx.) | Status |
|-----------|-----------------|--------|
| **DEV-Level 1** (*ca.* current) | Modular TCP/IP Servers Service · Linux + FreeBSD platform compat · `max25d` / `max25d0` · platform detection · INI examples · tests · rootless foundation | **active** |
| **DEV-Level 2** (*ca.*) | Main + Secondary supervision · cross-host wiring · `max25-tun` sidecar · deeper Linux/FreeBSD integration extras | **planned** |
| **DEV-Level 3** (*ca.*) | WebSocket gateway / browser terminal · other still-open mid-tier items (**not** CRDOP expansion) | **planned** |
| **DEV-Level 4** (*ca.*) | CRDOP expansion — OSS polish, modulation milestones, deeper integration | **deferred** |
| **Later** | AI / assistant integration · like-features — explicitly deferred; separate commit track | **deferred** |

### Later (explicitly deferred)

- **AI / assistant integration** and **like-features** — documented only; implementation and git commits wait for a separate user decision (likely beyond DEV-Level 4 or at end of current roadmap arc)

## Mandatory goals (non-negotiable)

| Goal | Requirement |
|------|-------------|
| **No root for day-to-day operation** | Bringing BayCom (and the full MAX25 stack) up, down, and on-air must **not** require `sudo` or a root shell for the operator |
| **One-time setup only** | Privileged steps (module install, `/etc` templates, UART probe) happen **once** at install or first-run — not on every boot |
| **Automatic setup** | Guided or scripted first-run: detect device class, write INI, enable services — minimal manual editing |
| **Approachable, not bloated** | No enterprise installer circus — systemd units, group membership, clear `max25-ctl` verbs, sensible defaults |
| **Experienced users welcome** | Power users can still edit INI, run multiple profiles, attach HyBBX — automation is the default path, not a cage |

These goals apply to **all v1-active device paths** on Linux (`tnc2c`, `baycom-ser12`, `soft-crdop`), with **BayCom kernel-ser12** as the hardest case (kernel module + UART).

## v1 limitation (honest)

v1.0.0 still calls `sudo baycom-pr-ctl` from `max25-ctl` and documents root for module load, `setserial`, and site INI under `/etc/baycom/`. That is **accepted technical debt** until v2 — not the long-term model.

## Target operator experience (v2)

```
One-time (admin or package install):
  install-max25.sh --system   OR   .deb / image flash
  → modules, udev, systemd units, /etc templates, dialout group hint

Every day (normal user, no sudo):
  max25-ctl start
  max25-terminal
```

Comparable mental model for Windows-experienced users: **install once like a driver**, then **start the app** — not “open terminal as administrator every time”.

## BayCom without root — design direction

Kernel `baycom_ser_fdx` still needs kernel privileges **somewhere**; v2 moves that to **boot/install time**, not the operator session.

| Layer | v1 (today) | v2 (target) |
|-------|------------|-------------|
| Module load | `sudo modprobe` per start | `modules-load.d` + packaged kmod at install |
| UART iobase/IRQ | `sudo baycom-pr-ctl setup` | Cached in site INI at first-run; optional udev by-id |
| Stack start | `sudo baycom-pr-ctl start` | `max25-baycom.service` (system) or user socket activation |
| KISS PTY | root-created | owned by `max25` group, world-readable path documented |
| `max25d` | often `sudo max25d` | user service or `max25` system user |
| Operator command | `sudo max25-ctl start …` | **`max25-ctl start`** as login user |

**BayCom on kernel-ser12 hosts:** `bcsf0` stays unchanged. TUN **`max25d0`** (when enabled) is an additional IP path via KISS PTY ([NETDEV.md](NETDEV.md)).

v2 targets [Main + optional Secondaries](ARCHITECTURE.md#host-layout--main--secondaries) on one server.

### USB / async BayCom (easier path)

`baycom-kiss` and USB-serial KISS already need only `dialout` — v2 should **prefer auto-detection** of USB vs onboard UART and pick the no-kernel path when appropriate.

## Automatic setup (v2 — proportionate)

Planned building blocks (not all required for first v2 tag):

| Building block | Purpose |
|----------------|---------|
| `max25-ctl setup` | Interactive or `--yes` non-interactive first-run wizard |
| `install-max25.sh --system` | One sudo boundary: modules, units, `/etc/max25`, groups |
| systemd | `max25d.service`, `max25-baycom.service`, `max25-tun.service` (when netdev lands) |
| shipped udev rules | `dialout` + stable `/dev/serial/by-id/` symlinks |
| Pi / homeserver profile | Defaults for common UART disable-console hint, single-modem INI |
| optional Web setup page | Later — not a blocker if CLI wizard is solid |

**Out of scope for v2:** full GUI desktop installer, cloud account, container-only distribution.

## Acceptance (v2 release gates)

- [ ] Operator on clean Pi/homeserver image brings stack up **without sudo** after one-time install
- [ ] BayCom kernel-ser12: `max25-ctl start --hardware modems --device baycom-ser12` as normal user
- [ ] TNC2C and `soft-crdop` paths also rootless for day-to-day
- [ ] Documented first-run under 15 minutes for a prepared image
- [ ] `./scripts/release-check.sh` extended with rootless smoke where feasible (mock / CI)
