# MainAX25-Stack v1.0.0 Scope

**Date:** 2026-07-12 · **Target version:** `1.0.0-rc1` → `1.0.0`

MainAX25 (MAX25) is the **modem/TNC owner** — standalone-first, HyBBX-ready. HyBBX (external) attaches to a **running** stack; it does not replace boot-wait, kernel BayCom lifecycle, or KISS PTY creation.

## Goal

Ship a **low-effort, automatic v1.0.0** that:

1. Lets an operator bring up **one radio path** without HyBBX (standalone).
2. Exposes a **clear attach contract** for HyBBX Secondary (INI examples + lifecycle hooks).
3. Passes **offline CI gates** (`make test`, `scripts/release-check.sh`) on Linux.

Aggressive scope cut: no minicom fork, no BSD port, no PK-TNC2 until hardware, no scaffold device promotion beyond documented templates.

### What HyBBX needs from MAX25

HyBBX does **not** vendor MainAX25. It needs the operator (or automation) to:

| HyBBX plugin | MAX25 preparation | HyBBX opens |
|--------------|-------------------|-------------|
| `packet_radio` | `tnc2c-boot-wait.sh` (DTR/power-on) | Serial KISS after boot-wait OK |
| `baycom` | `baycom-pr-ctl preflight` + `start` | Kernel netdev or KISS PTY |
| `crdop` | `crdopc` running (TCP :8515) | TCP control/data ports |

MAX25 owns everything HyBBX does **not** provide: TNC boot/DTR, kernel module load, KISS PTY, `axports` sync.

## Operator workflows

### Standalone (default v1 path)

**TNC2C (serial AX.25):**

```bash
make all
./scripts/max25-ctl start --hardware tncs --device tnc2c   # power-cycle TNC during boot-wait
./stacks/tncs/tnc2c-integration-test.sh                      # optional, with hardware
# Terminal clients: kissattach / listen / HyBBX Main local TNC
```

**BayCom SER12 (kernel modem):**

```bash
make all
sudo make -C stacks/baycom-pr install    # once
sudo ./stacks/baycom-pr/scripts/baycom-pr-ctl preflight
sudo ./stacks/baycom-pr/scripts/baycom-pr-ctl start
sudo ./stacks/baycom-pr/scripts/baycom-pr-ctl listen radio
```

**CRDOP soft modem (ARDOP, not AX.25):**

```bash
make all
./scripts/max25-ctl start --hardware soft-modems --device soft-crdop
# crdopc on TCP :8515 — parallel to packet radio
```

### HyBBX-edge workflow

1. Run MAX25 device prep (boot-wait or `baycom-pr-ctl start` or `crdopc`).
2. Merge matching `share/hybbx/*-edge.ini.example` into Secondary `hybbx.ini`.
3. Build HyBBX with required plugin (`packet_radio` default; `-DHYBBX_PLUGIN_BAYCOM=ON` or `CRDOP=ON`).
4. Start HyBBX Secondary → HBX link to Main.

**Order rule:** MAX25 stack **before** HyBBX transport plugin. Never share serial port between minicom and HyBBX.

## Success criteria / acceptance tests

### Offline (CI / release-check)

| Gate | Command | Pass condition |
|------|---------|----------------|
| Build | `make all` | exit 0; binaries present |
| BayCom QA | `make -C stacks/baycom-pr test` | `test-all passed` |
| CRDOP smoke | `make -C stacks/crdop smoke` | version smoke OK |
| Plugin registry | `make plugins` | ≥12 `plugin.yaml`; manifest valid |
| Discovery JSON | `discover-plugins.sh --json` | all v1 active devices listed |
| Umbrella gate | `make release-check` | all checks green |
| HyBBX INI refs | release-check | all `share/hybbx/*.example` exist |

### Hardware (manual, pre-`v1.0.0` tag)

| Device | Acceptance |
|--------|------------|
| **tnc2c** | Boot-wait OK → `tnc2c-integration-test.sh` passes → KISS frames on air (or loopback) |
| **baycom-ser12** | `baycom-pr-ctl selftest` or `listen`/`call` on configured port |
| **soft-crdop** | `crdopc` accepts TCP; HyBBX `crdop` plugin connects (RF optional for v1.0.0-rc) |

### HyBBX integration (manual)

| Path | Acceptance |
|------|------------|
| TNC2C edge | Secondary `packet_radio` links to Main after boot-wait |
| BayCom edge | Secondary `baycom` sees `bcsf0` or KISS PTY after stack start |
| CRDOP edge | Secondary `crdop` connects to `127.0.0.1:8515` |
